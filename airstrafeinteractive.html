Game Feel Pipeline

A detailed look into my process of adding a screenshake when a tree collapses onto the ground.

Initial tree falling, pre-screenshake

Step 1: Identifying places for polish

The process starts with me analyzing which areas of the game could use some extra game feel. I found that chopping down a tree was unsatisfying and could use more game feel to create a more satisfying experience.

Cutting down a tree has a long buildup of multiple swings, so the final tree collapse should be sufficiently juicy to reward players for their hard work.

Solution using the dot product of the tree's up vector & the vector from the tree center point to the collision point

Step 2: Initial testing & problem-solving

My first course of action was to trigger the screenshake when the tree's collider connected with the ground. This didn't work as the bottom of the tree would touch the ground first, resulting in a screenshake before the tree had finished collapsing to the ground.

To fix this, I had to find a way to detect when the entire length of the tree collided with the ground. I decided to check if the top half of the tree collided with the ground, using collision points and checking if the point of collision was on the top half of the tree's collider.

Code implementation (pre-serialization of screen shake values)

Step 3: Implementation

I tested multiple equations for detecting if the collision point was in the top half of the tree and decided on using a dot product formula. If the dot product of the tree's up vector and the vector from the tree's center to the point of collision was positive, the collision point is on the top half of the tree.

Final tree fall with screenshake

Step 4: Screenshake!

I called DOTween's ShakePosition function through my ScreenShakeManager, using a bool check to make sure it only triggered once. I serialized all the values for easy tweaking in the Inspector. The final result was consistent regardless of the tree's angle of fall.
